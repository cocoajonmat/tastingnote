## 💡 토론 중 나온 아이디어 (친구와 논의 필요)

### Discovery (발견) 기능
> 기존 검색이 "찾고 싶은 걸 알고 있을 때" 위주라면,
> Discovery는 "몰랐던 걸 우연히 발견하는" 경험을 제공.

**아이디어 예시**
- 술 카테고리별 탐색 (예: "아이리쉬 위스키 중 별점 4점 이상 노트들")
- 향/맛 기반 탐색 (예: "바닐라 향이 난다고 기록한 노트들")
- 같은 술을 마신 다른 사람들의 노트 모아보기

**기대 효과**
- 공개 피드가 단순 타임라인을 넘어서 콘텐츠로서 가치 생김
- 유저가 새로운 술을 발견하는 계기가 됨

**논의 필요 사항**
- 이 기능을 넣을 것인지 여부
- 넣는다면 검색 탭 안에 포함할지, 별도 탐색 페이지를 만들지

## 🙋 친구와 상의할 질문 목록

---

**Q1. 이 서비스의 주 타겟이 누구인가?**
> 진지하게 술을 기록하는 사람(위스키 애호가, 와인 덕후)인지, 가볍게 공유하고 싶은 사람(인스타 감성)인지에 따라 UI/UX 방향이 완전히 달라짐.
> 예: 진지한 기록러라면 필드가 많아도 괜찮지만, 가벼운 공유 중심이라면 입력 항목을 최소화하고 사진 중심으로 가야 함.

---

**Q2. 이상적인 노트 작성 시간이 얼마나 되는가?**
> 술 마시고 취한 상태에서 노트를 작성한다고 가정했을 때, 현실적으로 몇 분을 쓸 수 있을까?
> 지금 Note 필드가 꽤 많음 (술 이름, 별점, taste, aroma, pairing, description, location, 날짜, 태그, 사진, 공개여부).
> "30초짜리 빠른 기록"을 원한다면 필수 필드를 최소화하고 나머지는 나중에 추가할 수 있게 해야 함.
> "5분짜리 상세 기록"을 원한다면 지금 설계 그대로도 괜찮음.

---

**Q3. Discovery(발견) 기능을 넣을 건지?**
> 지금 검색은 "찾고 싶은 걸 알고 있을 때" 위주 (태그, 노트 제목, 내용 검색).
> Discovery는 "몰랐던 걸 우연히 발견하는" 경험 — 예: "아이리쉬 위스키 중 별점 4점 이상 노트들", "바닐라 향이 난다고 기록한 노트들", "이 술을 마신 다른 사람들 노트 모아보기".
> 넣는다면 검색 탭 안에 포함할지, 별도 탐색 페이지를 만들지도 결정 필요.

---

**Q4. 서비스 초반 유저가 적을 때 피드를 어떻게 채울 건지? ✅ 확정**
> ✅ 확정 — 관리자가 초반 노트 10~20개 직접 작성 (A안) + 가입 직후 첫 노트 작성 유도 (Q6 B안) 조합
> - 피드가 비어있어도 "내 기록 앱"이라는 인식을 온보딩에서 심어주면 이탈 방지 가능

---

**Q5. 술 DB를 어떻게 키울 건지? 크라우드소싱 도입 시점은? ✅ 확정**
> ✅ 확정 — 초기 데이터 SQL 삽입 + 크라우드소싱 조합
> - 1단계: SQL로 자주 마시는 술 초기 데이터 삽입
> - 2단계: 크라우드소싱(유저 등록 요청 → 관리자 승인)으로 DB 확장
> - 크라우드소싱을 Tag/Like보다 먼저 구현 (술 데이터 품질 = 서비스 기반)
> - 초반 운영: 어드민 페이지 없이 Swagger에서 관리자 API 호출
>
> **별칭(AlcoholAlias) 크라우드소싱 방식 확정**
> 유저가 술 등록 요청 시 공식 명칭 + 별칭을 함께 입력, 관리자가 검토 후 승인.
> → **B안(유저가 별칭도 함께 제안)으로 확정**. Vivino도 동일 방식.
>
> **관리자 처리 방식 확정 — 3가지 액션**
> - **승인 (신규 등록)**: 새로운 술 → Alcohol + Alias 생성
> - **별칭으로 병합**: 이미 있는 술인데 다른 이름으로 요청됨 → 기존 Alcohol에 alias만 추가
> - **거절**: 장난, 부적절한 요청 등
> 병합 기능의 효과: 거절당한 유저도 "내 요청이 별칭으로 반영됨" → 긍정적 경험 + 검색 품질 자연 향상
>
> **AlcoholRequest 테이블 설계**
> - status: PENDING / APPROVED / MERGED / REJECTED
> - mergedToAlcoholId: 별칭 병합 시 어떤 Alcohol에 붙였는지 기록
> - 목록 조회 시 similarAlcohols 포함 (기존 DB 유사 술 자동 검색 → 관리자 판단 도움)

---

**Q6. 가입 직후 유저에게 어떤 첫 경험을 줄 건지? (온보딩) ✅ 확정**
> ✅ 확정 — 가입 직후 "첫 노트 작성하기" 흐름으로 유도 (B안)
> - 이 서비스의 핵심 가치는 "내 기록"이므로 처음부터 노트 작성을 각인
> - 백엔드 구현: 로그인 응답에 isFirstLogin 필드 추가 → 프론트에서 분기 처리
> - 보완책: 관리자가 초반 노트 10~20개 직접 작성해 피드 채워두기 (A안 병행)

---

**Q7. 별점을 정수(1~5)로 유지할지, 0.5단위(1.0~5.0)로 할지? ✅ 확정**
> 0.5단위(1.0~5.0)로 확정. Vivino와 동일한 방식. DB 컬럼 DECIMAL(2,1), 코드 타입 Double로 변경 완료.

---

**Q8. 태그 품질을 어떻게 관리할 건지? ✅ 확정**
> 태그 자동완성에 사용 횟수 표시 방식으로 확정 (예: #위스키 (234)).
> 맛/향은 FlavorSuggestion 관리자 목록에서 **선택만 가능** (Vivino 방식) — NoteFlavor 중간 테이블로 구현 완료.
> 일반 태그는 자유입력 + 사용 횟수 표시.
> 사용 횟수는 NoteTag 개수 실시간 카운팅 방식 (Tag 테이블 컬럼 추가 없음).
> 맛/향을 선택 전용으로 한 이유: 자유 입력 시 "바닐라", "바닐라향", "vanilla" 등 같은 것이 다르게 저장되어 술 상세 페이지/Discovery 통계를 낼 수 없음.

**Q9. 프로필 페이지를 어떻게 구성할 건지? ✅ 확정**
> 팔로우 기능이 없는 상태에서 타인 프로필에 들어갔을 때 뭘 보여줄 건지.
> ✅ 확정 — 미니 포트폴리오 방식
> - 공개 노트 목록
> - 많이 마신 술 카테고리 (예: "위스키 애호가")
> - 평균 별점, 총 노트 수, 즐겨 쓰는 태그 Top 5
> - 별도 집계 테이블 없이 기존 데이터 쿼리로 구현

**Q10. 회원탈퇴 처리 방식을 어떻게 할 건지? ✅ 확정**
> ✅ 확정 (30일 유예 후 완전삭제 + 유예기간 중 연관 데이터 유지)
> - 탈퇴 시 deleted_at 기록 + 모든 노트 isPublic = false
> - 30일 유예 기간 후 Hard Delete 스케줄러로 완전 삭제
> - 유예 기간 중 계정 복구 가능
> - 탈퇴 시 연관 데이터(좋아요, 태그 등) 유지 → Hard Delete 시 함께 삭제
> - 이유: 복구 시 데이터 온전히 살아있어야 하고, 30일간 좋아요 카운트 약간 부정확해도 초반엔 큰 문제 없음

**Q11. 이미지 업로드 방식과 개수 제한은? ✅ 부분 확정**

**Q12. 알림 기능을 어떻게 구현할 건지? ✅ 부분 확정**
> 현재 전체 보류 상태. 추후 도입 시 방식 결정 필요.
> 서비스 내 알림(인앱)인지, 이메일 알림인지, 아니면 둘 다인지는 미확정.

> - 첫 번째 좋아요: "홍길동님이 회원님의 노트를 좋아합니다."
> - 이후 누적: "홍길동 외 5명이 회원님의 노트를 좋아합니다."
> - 같은 노트의 좋아요 알림은 하나로 묶어서 카운트만 올리는 방식 (알림 폭탄 방지)

> ✅ 혼용 방식으로 확정
> - 공개 피드: 커서 페이지네이션 (무한 스크롤)
>   - 실시간으로 새 노트가 올라오는 피드 특성상 페이지 번호 방식은 중복/누락 문제 발생 가능

>   - 내 노트는 추가 속도가 느려 중복/누락 문제 거의 없고, 특정 시점 노트를 찾을 때 페이지 번호가 훨씬 편함

> ✅ 확정 — 아래 형식으로 통일, ApiResponse<T> 공통 클래스로 구현 완료

> // 성공
> { "success": true, "data": { ... } }
>
> // 실패
> { "success": false, "message": "에러 메시지" }
> ```

**Q15. Like 리액션 타입을 몇 가지로 할 건지? ✅ 확정**
> ✅ 확정 — CHEERS 단일 타입
> - Untappd의 Toast(건배)와 동일 방식. 술 앱 정체성이 담겨 있고 단순해서 engagement 높음

> - WANT("마셔보고 싶다")는 나중에 Alcohol 상세 페이지의 위시리스트 기능으로 분리 예정

**Q16. 술(Alcohol) 상세 페이지를 만들 건지?**
> 현재 설계에는 술 자체 페이지가 없음 — 술은 노트 작성 시 선택하는 용도로만 사용됨.
> Vivino는 술 페이지가 핵심 콘텐츠 — 평균 별점, 다른 사람 노트, 맛/향 분포가 술 페이지에 집약됨.

**Q20. N+1 쿼리 문제를 해결해야 하는가?** ⭐⭐⭐⭐⭐

> **문제 발견**
> NoteResponse.from(note)에서 note.getUser(), note.getAlcohol()을 호출함.
> 둘 다 FetchType.LAZY라서, 피드에서 노트 50개를 조회하면:
> - 노트 50개 쿼리 1번
> - 각 노트마다 user 조회 50번 + alcohol 조회 50번
> → 총 101번 쿼리 발생 (N+1 문제)
> 지금은 데이터가 적어서 체감이 없지만 피드에 노트가 100개만 되어도 응답이 느려짐.
>
> **왜 LAZY 로딩을 썼나**
> LAZY 로딩은 연관 데이터를 "실제로 쓸 때만" DB에서 가져오는 방식.
> 예: note.getUser()를 코드에서 안 부르면 users 테이블을 아예 조회 안 함.
> 단건 조회처럼 연관 데이터가 필요 없는 경우에는 EAGER보다 훨씬 효율적.
> 그래서 JPA 권장 설정이기도 함.
>
> **문제는 목록 조회**
> 단건 조회와 달리 목록 조회는 루프를 돌면서 각 note마다 getUser()를 호출하게 됨.
> 결국 노트 수만큼 쿼리가 터지는 구조 — LAZY 로딩의 의도치 않은 부작용.
>
> **왜 EAGER로 바꾸면 안 되나**
> EAGER로 바꾸면 note 하나를 조회할 때도 항상 user와 alcohol을 JOIN해서 가져옴.
> 예: 단건 조회, 존재 여부 확인 등 user가 불필요한 상황에서도 항상 JOIN 발생 → 오히려 더 비효율.
> "필요할 때 필요한 곳에서만" 해결해야 함.
>
> **해결: @EntityGraph**
> Repository의 목록 조회 메서드에만 @EntityGraph를 붙여서 해당 쿼리에서만 JOIN으로 한 번에 가져오도록 지정.
> LAZY 설정은 그대로 유지하면서, 목록 조회 시에만 동작 방식을 바꾸는 것.
> 결과: 101번 쿼리 → 1번으로 감소.
>
> **JOIN FETCH와의 차이**
> JOIN FETCH는 JPQL에 직접 작성하는 방식 — 더 유연하지만 쿼리를 직접 써야 함.
> @EntityGraph는 어노테이션으로 선언하는 방식 — 코드가 깔끔하고 Spring Data JPA 메서드 이름 쿼리와 함께 쓰기 좋음.
> 이 프로젝트에서는 @EntityGraph를 선택할 예정.
>
> **면접 예상 질문과 답변**
> Q: "왜 처음부터 EAGER로 안 했나요?"
> A: "EAGER는 항상 JOIN을 강제해서 불필요한 경우에도 부하가 생깁니다. LAZY를 기본으로 유지하고, 목록 조회처럼 연관 데이터가 반드시 필요한 곳에만 @EntityGraph로 한정해서 해결하는 게 더 정교한 접근입니다."

**Q21. 검색 품질 개선 — LIKE에서 Full-Text Search로** ⭐⭐⭐⭐⭐

> **현재 방식의 한계**
> `LIKE %키워드%` 방식은 두 가지 문제가 있음:
> 1. 성능: 앞에 %가 붙으면 인덱스를 못 써서 데이터가 많아질수록 전체 테이블을 스캔함
> 2. 품질: 모든 결과를 동등하게 취급 — "블랙라벨"을 검색했을 때 정확히 일치하는 술이나 별칭에 "블랙"이 포함된 술이나 순서가 같음. 어떤 게 먼저 나올지 예측 불가.
>
> **Full-Text Search가 다른 이유**
> DB가 각 단어를 쪼개서 별도의 인덱스(역색인)를 미리 만들어 놓는 방식.
> 예: "조니워커 블랙라벨" → "조니워커", "블랙라벨" 각각 인덱스 생성
> 검색 시 이 인덱스를 바로 조회하므로 LIKE보다 훨씬 빠름.
> 또한 각 결과에 relevance score(관련도 점수)를 부여 — 정확히 일치할수록 점수가 높아서 상위 노출 가능.
>
> **실제 서비스 비교**
> - Vivino: 와인 이름 정확 일치 > 생산자 이름 일치 > 부분 일치 순으로 정렬 + 리뷰 수 많은 와인 상위 노출
> - Untappd: 맥주 이름 정확 일치 우선 + 체크인 수 반영
> 둘 다 단순 포함 여부가 아니라 "얼마나 관련 있는가"로 정렬함.
>
> **전환 시점**
> 지금은 술 DB가 수백 개 수준이라 LIKE로도 충분함.
> 데이터가 수천 개를 넘거나 검색 결과 순서에 대한 불만이 생길 때 전환 예정.
> MySQL 8.0 이상에서는 별도 설치 없이 FULLTEXT 인덱스 추가만으로 바로 사용 가능.
>
> **면접 예상 질문과 답변**
> Q: "처음부터 Full-Text Search를 쓰지 않은 이유가 있나요?"
> A: "초반에는 술 DB가 작아서 LIKE로도 성능 문제가 없었고, 오버엔지니어링을 피하고 싶었습니다. 단순한 방식으로 시작해서 실제로 한계를 느끼는 시점에 전환하는 게 더 실용적이라고 판단했습니다. 전환 경로도 LIKE → MySQL Full-Text Search → 필요하면 Elasticsearch 순으로 단계적으로 계획해뒀습니다."

**Q22. 동시 편집 문제 — 낙관적 락(Optimistic Locking)** ⭐⭐⭐⭐

> **문제 발견**
> 같은 노트를 폰과 노트북에서 동시에 열고, 둘 다 수정 후 저장하면?
> 지금은 나중에 저장한 쪽이 먼저 저장한 내용을 조용히 덮어씀 (Last Write Wins).
> 유저 입장에서는 내가 분명히 저장했는데 내용이 사라진 상황 — 신뢰도 하락.
>
> **락(Lock)이란**
> 여러 곳에서 동시에 같은 데이터를 수정하려 할 때 충돌을 방지하는 메커니즘.
> 크게 두 가지 방식이 있음:
>
> - 비관적 락(Pessimistic Lock): "누군가 수정 중이면 아예 접근 못 하게 잠금"
>   → DB에 SELECT FOR UPDATE로 레코드 자체를 잠금. 다른 사람은 수정 완료될 때까지 대기.
>   → 충돌이 자주 일어나는 환경에 적합하지만, 대기 시간 발생 + 데드락 위험.
>
> - 낙관적 락(Optimistic Lock): "충돌이 드물다고 낙관적으로 가정하고, 저장할 때 확인"
>   → 잠금 없이 수정 허용. 저장 직전에 "내가 읽었을 때랑 지금이랑 같은 상태인가?" 확인.
>   → 다르면 충돌로 판단하고 에러 반환.
>
> **왜 낙관적 락을 선택하나**
> 노트 편집은 동시 충돌이 드문 케이스 — 같은 노트를 두 기기에서 동시에 수정하는 경우가 많지 않음.
> 비관적 락은 충돌이 거의 없는데도 항상 DB 잠금을 걸어서 성능 낭비.
> 낙관적 락은 평상시엔 아무 부하 없이 작동하다가 진짜 충돌이 생겼을 때만 처리함.
>
> **동작 방식 (@Version)**
> Note 엔티티에 version 컬럼 추가. 수정할 때마다 version이 1씩 올라감.
> 저장 시 "내가 읽었을 때 version"과 "현재 DB의 version"을 비교.
> 다르면 → 그 사이 누군가 수정했다는 뜻 → OptimisticLockException 발생 → "다른 곳에서 수정됨, 새로고침 필요" 안내.
>
> **면접 예상 질문과 답변**
> Q: "비관적 락과 낙관적 락의 차이를 알고 있나요? 왜 낙관적을 선택했나요?"
> A: "비관적 락은 충돌을 아예 막는 방식이고, 낙관적 락은 충돌을 감지해서 처리하는 방식입니다. 노트 편집은 같은 노트를 두 곳에서 동시에 수정하는 경우가 드물어서, 항상 DB 잠금을 거는 비관적 락은 오버헤드가 크다고 판단했습니다. 충돌 빈도가 낮고 사용자 경험(대기 없음)이 중요한 경우엔 낙관적 락이 더 적합합니다."

**Q23. 술 검색 API에 캐싱이 필요한가?** ⭐⭐⭐

> **문제 발견**
> Alcohol 테이블은 관리자만 수정할 수 있고 변경 빈도가 극히 낮음 (하루에 몇 건 수준).
> 근데 유저가 노트 작성할 때마다 술 이름 자동완성 API를 호출 → 매번 DB 쿼리 실행.
> "위스키"로 검색하는 유저가 100명이면 똑같은 쿼리를 100번 실행하는 셈.
>
> **캐싱이란**
> 자주 요청되는 데이터를 메모리(RAM)에 저장해두고, 같은 요청이 오면 DB까지 가지 않고 메모리에서 바로 꺼내주는 방식.
> DB 접근: 수십 ms / 메모리 접근: 수 μs (수천 배 빠름)
>
> **어떤 데이터에 적합한가**
> 캐싱은 읽기가 많고 변경이 적은 데이터에 효과적.
> - 적합: 술 정보, 맛/향 제안 목록(FlavorSuggestion), 카테고리 목록 → 거의 안 바뀜
> - 부적합: 노트 목록, 피드 → 실시간으로 계속 바뀜
>
> **캐시 무효화 문제**
> 캐싱의 핵심 어려움은 "언제 캐시를 지울 것인가"임.
> 술 정보가 수정됐는데 캐시가 그대로면 유저에게 옛날 정보를 보여주게 됨.
> 해결: 관리자가 술 정보를 수정/추가할 때 해당 캐시를 명시적으로 삭제(@CacheEvict).
> 지금은 관리자 API가 없으니 나중에 함께 구현 예정.
>
> **단계적 확장 계획**
> 1단계: Spring Cache (로컬 메모리 캐시) — 서버 1대일 때 충분
> 2단계: Redis 캐시 — 서버가 여러 대로 늘어나면 로컬 메모리 캐시는 서버마다 달라서 문제 발생. Redis는 별도 서버에서 공유 캐시 역할.
>
> **면접 예상 질문과 답변**
> Q: "캐시를 도입했을 때 생길 수 있는 문제는 무엇인가요?"
> A: "캐시 무효화 문제입니다. 원본 데이터가 바뀌었을 때 캐시가 그대로면 오래된 데이터를 보여주게 됩니다. 이를 해결하기 위해 술 정보 수정 시 @CacheEvict로 관련 캐시를 삭제하도록 구현했습니다. 또한 서버가 여러 대로 늘어나면 로컬 메모리 캐시는 서버마다 달라지는 문제가 있어서 Redis로 전환하는 것도 고려하고 있습니다."
