## 💡 토론 중 나온 아이디어 (친구와 논의 필요)

### Discovery (발견) 기능
> 기존 검색이 "찾고 싶은 걸 알고 있을 때" 위주라면,
> Discovery는 "몰랐던 걸 우연히 발견하는" 경험을 제공.

**아이디어 예시**
- 술 카테고리별 탐색 (예: "아이리쉬 위스키 중 별점 4점 이상 노트들")
- 향/맛 기반 탐색 (예: "바닐라 향이 난다고 기록한 노트들")
- 같은 술을 마신 다른 사람들의 노트 모아보기

**기대 효과**
- 공개 피드가 단순 타임라인을 넘어서 콘텐츠로서 가치 생김
- 유저가 새로운 술을 발견하는 계기가 됨

**논의 필요 사항**
- 이 기능을 넣을 것인지 여부
- 넣는다면 검색 탭 안에 포함할지, 별도 탐색 페이지를 만들지

## 🙋 친구와 상의할 질문 목록

---

**Q1. 이 서비스의 주 타겟이 누구인가?**
> 진지하게 술을 기록하는 사람(위스키 애호가, 와인 덕후)인지, 가볍게 공유하고 싶은 사람(인스타 감성)인지에 따라 UI/UX 방향이 완전히 달라짐.
> 예: 진지한 기록러라면 필드가 많아도 괜찮지만, 가벼운 공유 중심이라면 입력 항목을 최소화하고 사진 중심으로 가야 함.

---

**Q2. 이상적인 노트 작성 시간이 얼마나 되는가?**
> 술 마시고 취한 상태에서 노트를 작성한다고 가정했을 때, 현실적으로 몇 분을 쓸 수 있을까?
> 지금 Note 필드가 꽤 많음 (술 이름, 별점, taste, aroma, pairing, description, location, 날짜, 태그, 사진, 공개여부).
> "30초짜리 빠른 기록"을 원한다면 필수 필드를 최소화하고 나머지는 나중에 추가할 수 있게 해야 함.
> "5분짜리 상세 기록"을 원한다면 지금 설계 그대로도 괜찮음.

---

**Q3. Discovery(발견) 기능을 넣을 건지?**
> 지금 검색은 "찾고 싶은 걸 알고 있을 때" 위주 (태그, 노트 제목, 내용 검색).
> Discovery는 "몰랐던 걸 우연히 발견하는" 경험 — 예: "아이리쉬 위스키 중 별점 4점 이상 노트들", "바닐라 향이 난다고 기록한 노트들", "이 술을 마신 다른 사람들 노트 모아보기".
> 넣는다면 검색 탭 안에 포함할지, 별도 탐색 페이지를 만들지도 결정 필요.

---

**Q4. 서비스 초반 유저가 적을 때 피드를 어떻게 채울 건지?**
> 지금 피드 정렬이 "좋아요 많은 순 / 최신순"인데, 초반엔 유저도 없고 좋아요도 없어서 피드가 텅 빔.
> 텅 빈 피드를 본 신규 유저는 그냥 나가버릴 가능성이 높음 (콜드 스타트 문제).
> 해결책 예시: 관리자가 초반 노트를 직접 몇 개 작성해두기, 추천 노트 고정 배너, 가입 직후 노트 작성 유도 등.

---

**Q5. 술 DB를 어떻게 키울 건지? 크라우드소싱 도입 시점은?**
> 관리자 둘이서 술 DB를 직접 채우는 데는 한계가 있음. 한국에서 마실 수 있는 술 종류가 너무 많음.
> DB가 빈약하면 대부분의 노트가 텍스트 직접 입력으로 채워지고, 같은 술이 여러 이름으로 저장되는 문제 발생.
> 크라우드소싱(유저 등록 요청 → 관리자 승인)을 처음부터 넣을지, 나중에 추가할지 결정 필요.
> 초반엔 어드민 페이지 없이 DB 툴로 직접 승인해도 충분히 운영 가능.

---

**Q6. 가입 직후 유저에게 어떤 첫 경험을 줄 건지? (온보딩)**
> 새로 가입한 유저가 처음 앱을 열면 피드도 비어있고, 내 노트도 없고, 술 DB도 빈약한 상태.
> 이 상태에서 유저가 "뭐 하는 앱이지?" 하고 그냥 나가버릴 수 있음.
> 잘 만든 서비스들은 가입 직후 바로 행동을 유도함: Untappd는 "첫 체크인 해보세요!", Vivino는 "좋아하는 와인 스타일 선택" 등.
> TastingNote는 기록 중심이니 "첫 노트 바로 작성하기" 흐름을 온보딩에 넣는 게 좋을 수 있음.

---

**Q7. 별점을 정수(1~5)로 유지할지, 0.5단위(1.0~5.0)로 할지? ✅ 확정**
> 0.5단위(1.0~5.0)로 확정. Vivino와 동일한 방식. DB 컬럼 DECIMAL(2,1), 코드 타입 Double로 변경 완료.

---

**Q8. 태그 품질을 어떻게 관리할 건지? ✅ 확정**
> 태그 자동완성에 사용 횟수 표시 방식으로 확정 (예: #위스키 (234)).
> 맛/향은 FlavorSuggestion 관리자 목록에서 선택, 일반 태그는 자유입력 + 사용 횟수 표시 (Vivino 방식).
> 사용 횟수는 NoteTag 개수 실시간 카운팅 방식 (Tag 테이블 컬럼 추가 없음).

---

**Q9. 프로필 페이지를 어떻게 구성할 건지? ✅ 확정**
> 팔로우 기능이 없는 상태에서 타인 프로필에 들어갔을 때 뭘 보여줄 건지.
> ✅ 확정 — 미니 포트폴리오 방식
> - 공개 노트 목록
> - 많이 마신 술 카테고리 (예: "위스키 애호가")
> - 평균 별점, 총 노트 수, 즐겨 쓰는 태그 Top 5
> - 별도 집계 테이블 없이 기존 데이터 쿼리로 구현

---

**Q10. 회원탈퇴 처리 방식을 어떻게 할 건지? ✅ 확정**
> ✅ 확정 (30일 유예 후 완전삭제 + 유예기간 중 연관 데이터 유지)
> - 탈퇴 시 deleted_at 기록 + 모든 노트 isPublic = false
> - 30일 유예 기간 후 Hard Delete 스케줄러로 완전 삭제
> - 유예 기간 중 계정 복구 가능
> - 탈퇴 시 연관 데이터(좋아요, 태그 등) 유지 → Hard Delete 시 함께 삭제
> - 이유: 복구 시 데이터 온전히 살아있어야 하고, 30일간 좋아요 카운트 약간 부정확해도 초반엔 큰 문제 없음

---

**Q11. 이미지 업로드 방식과 개수 제한은? ✅ 부분 확정**
> - 저장 위치: AWS S3 ✅ 확정
> - 업로드 시점: 노트 저장 요청 시 이미지도 함께 S3에 업로드 ✅ 확정
>   - 이미지를 먼저 올려두는 방식은 노트 저장 안 하면 S3에 쓸모없는 이미지가 쌓이는 문제가 있어서 제외
> - 이미지 장수: 우선 1장으로 운영 (미확정, 추후 변경 가능)

---

**Q12. 알림 기능을 어떻게 구현할 건지? ✅ 부분 확정**
> 현재 전체 보류 상태. 추후 도입 시 방식 결정 필요.
> 서비스 내 알림(인앱)인지, 이메일 알림인지, 아니면 둘 다인지는 미확정.
> 인앱 알림을 실시간으로 하려면 WebSocket 또는 SSE 기술이 추가로 필요함.
>
> ✅ 좋아요 알림 방식 확정 — 페이스북 초기 방식
> - 첫 번째 좋아요: "홍길동님이 회원님의 노트를 좋아합니다."
> - 이후 누적: "홍길동 외 5명이 회원님의 노트를 좋아합니다."
> - 같은 노트의 좋아요 알림은 하나로 묶어서 카운트만 올리는 방식 (알림 폭탄 방지)

---

**Q13. 페이지네이션 방식을 어떻게 할 건지? ✅ 확정**
> 피드와 노트 목록을 나눠서 불러올 때 방식 결정 필요.
> ✅ 혼용 방식으로 확정
> - 공개 피드: 커서 페이지네이션 (무한 스크롤)
>   - 실시간으로 새 노트가 올라오는 피드 특성상 페이지 번호 방식은 중복/누락 문제 발생 가능
> - 내 노트 목록: 오프셋 페이지네이션 (페이지 번호)
>   - 내 노트는 추가 속도가 느려 중복/누락 문제 거의 없고, 특정 시점 노트를 찾을 때 페이지 번호가 훨씬 편함

---

**Q14. API 응답 형식을 어떻게 통일할 건지? ✅ 확정**
> 지금은 API마다 응답 형식이 제각각이라 프론트 개발자(친구)가 매번 "이 API는 에러가 어떻게 오지?" 확인해야 하는 상황.
> ✅ 확정 — 아래 형식으로 통일, ApiResponse<T> 공통 클래스로 구현 완료
> ```json
> // 성공
> { "success": true, "data": { ... } }
>
> // 실패
> { "success": false, "message": "에러 메시지" }
> ```
