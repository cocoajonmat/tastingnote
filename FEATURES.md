## 🎯 면접 스토리 — 앞으로 적용할 기술 업그레이드

> "처음엔 A로 시작했는데 한계를 느껴서 B로 전환한 경험"을 만들기 위한 계획.
> 별점은 면접 가치 기준.

| 순서 | 주제 | 현재 → 목표 | 면접 가치 |
|------|------|------------|-----------|
| 1 | N+1 쿼리 해결 | LAZY 로딩 → @EntityGraph | ⭐⭐⭐⭐⭐ |
| 2 | 검색 품질 개선 | LIKE → Full-Text Search + 정렬 | ⭐⭐⭐⭐⭐ |
| 3 | 낙관적 락 | Last Write Wins → @Version | ⭐⭐⭐⭐ |
| 4 | 캐싱 도입 | 매번 DB 조회 → Spring Cache | ⭐⭐⭐ |

자세한 내용은 Q20~Q23 참고.

## 🏆 면접 스토리 — 이미 적용 완료한 기술 경험

> "버그나 한계를 직접 발견해서 고친 경험"으로 이야기할 수 있는 항목들.
> 추상적인 지식이 아니라 "내 코드에서 이런 문제가 있었고 이렇게 해결했다"는 구체적인 스토리.

| 주제 | 경험 요약 | 면접 가치 | 세션 |
|------|----------|-----------|------|
| Refresh Token Rotation + Reuse Detection | JWT 재발급 시 hard delete 방식이었는데, 탈취 감지가 불가능하다는 걸 발견. OAuth 2.0 Security BCP 표준 패턴(revoked 플래그 + 재사용 감지 시 전체 무효화)으로 전환 | ⭐⭐⭐⭐⭐ | 11회차 |
| BigDecimal 정밀도 전환 | 별점 0.5 단위 검증을 Double + Math.round로 했는데 부동소수점 오차로 3.5001 같은 값이 간헐적으로 통과하는 것 발견. DB 컬럼이 DECIMAL인데 Java가 Double인 타입 불일치도 인지. BigDecimal + remainder 방식으로 재작성 | ⭐⭐⭐⭐ | 11회차 |
| TOCTOU race condition 방어 | 중복 신고 방지 로직이 existsBy + save 패턴이었고 DB unique 제약도 없어, 동시 요청 시 중복 신고가 조용히 쌓이는 상태. DB 제약을 진실의 원천으로 두고 DataIntegrityViolationException을 비즈니스 예외로 매핑하는 이중 구조 적용 | ⭐⭐⭐ | 11회차 |
| 상태 머신 설계 (Lifecycle ≠ Visibility) | NoteStatus(DRAFT/PUBLISHED)와 isPublic을 비슷하다고 엮어서 "공개 노트를 만들 방법이 없는" 버그가 발생. 두 필드가 의미적으로 다른 차원(라이프사이클 vs 공개 의도)이라는 걸 정리하고 분리 | ⭐⭐⭐ | 11회차 |
| 삭제 N+1 해결 | deleteAllByNoteId가 derived delete라 SELECT N + DELETE N이 발생하는 걸 확인. @Modifying + @Query로 단일 DELETE 쿼리로 전환 | ⭐⭐⭐ | 10회차 |
| Clickjacking 방어 | H2 콘솔용으로 X-Frame-Options를 disable 해둔 걸 발견, 운영 환경에서도 그대로 켜져 있어 보안 취약. 설정 제거 후 Spring Security 기본값(DENY) 복원 | ⭐⭐⭐ | 10회차 |

자세한 구현/설계 배경은 CHANGELOG.md 2026-04-11, 2026-04-12 엔트리와 LEARNING.md의 해당 섹션 참고.
11회차 항목의 면접 스토리 전체 흐름과 예상 질문/답변 스크립트는 **Q24~Q27**에서 확인 가능.

## 💡 토론 중 나온 아이디어 (친구와 논의 필요)

### Discovery (발견) 기능
> 기존 검색이 "찾고 싶은 걸 알고 있을 때" 위주라면,
> Discovery는 "몰랐던 걸 우연히 발견하는" 경험을 제공.

**아이디어 예시**
- 술 카테고리별 탐색 (예: "아이리쉬 위스키 중 별점 4점 이상 노트들")
- 향/맛 기반 탐색 (예: "바닐라 향이 난다고 기록한 노트들")
- 같은 술을 마신 다른 사람들의 노트 모아보기

**기대 효과**
- 공개 피드가 단순 타임라인을 넘어서 콘텐츠로서 가치 생김
- 유저가 새로운 술을 발견하는 계기가 됨

**논의 필요 사항**
- 이 기능을 넣을 것인지 여부
- 넣는다면 검색 탭 안에 포함할지, 별도 탐색 페이지를 만들지

## 🙋 친구와 상의할 질문 목록

---

**Q1. 이 서비스의 주 타겟이 누구인가?**
> 진지하게 술을 기록하는 사람(위스키 애호가, 와인 덕후)인지, 가볍게 공유하고 싶은 사람(인스타 감성)인지에 따라 UI/UX 방향이 완전히 달라짐.
> 예: 진지한 기록러라면 필드가 많아도 괜찮지만, 가벼운 공유 중심이라면 입력 항목을 최소화하고 사진 중심으로 가야 함.

---

**Q2. 이상적인 노트 작성 시간이 얼마나 되는가?**
> 술 마시고 취한 상태에서 노트를 작성한다고 가정했을 때, 현실적으로 몇 분을 쓸 수 있을까?
> 지금 Note 필드가 꽤 많음 (술 이름, 별점, taste, aroma, pairing, description, location, 날짜, 태그, 사진, 공개여부).
> "30초짜리 빠른 기록"을 원한다면 필수 필드를 최소화하고 나머지는 나중에 추가할 수 있게 해야 함.
> "5분짜리 상세 기록"을 원한다면 지금 설계 그대로도 괜찮음.

---

**Q3. Discovery(발견) 기능을 넣을 건지?**
> 지금 검색은 "찾고 싶은 걸 알고 있을 때" 위주 (태그, 노트 제목, 내용 검색).
> Discovery는 "몰랐던 걸 우연히 발견하는" 경험 — 예: "아이리쉬 위스키 중 별점 4점 이상 노트들", "바닐라 향이 난다고 기록한 노트들", "이 술을 마신 다른 사람들 노트 모아보기".
> 넣는다면 검색 탭 안에 포함할지, 별도 탐색 페이지를 만들지도 결정 필요.

---

**Q4. 서비스 초반 유저가 적을 때 피드를 어떻게 채울 건지? ✅ 확정**
> ✅ 확정 — 관리자가 초반 노트 10~20개 직접 작성 (A안) + 가입 직후 첫 노트 작성 유도 (Q6 B안) 조합
> - 피드가 비어있어도 "내 기록 앱"이라는 인식을 온보딩에서 심어주면 이탈 방지 가능

---

**Q5. 술 DB를 어떻게 키울 건지? 크라우드소싱 도입 시점은? ✅ 확정**
> ✅ 확정 — 초기 데이터 SQL 삽입 + 크라우드소싱 조합
> - 1단계: SQL로 자주 마시는 술 초기 데이터 삽입
> - 2단계: 크라우드소싱(유저 등록 요청 → 관리자 승인)으로 DB 확장
> - 크라우드소싱을 Tag/Like보다 먼저 구현 (술 데이터 품질 = 서비스 기반)
> - 초반 운영: 어드민 페이지 없이 Swagger에서 관리자 API 호출
>
> **별칭(AlcoholAlias) 크라우드소싱 방식 확정**
> 유저가 술 등록 요청 시 공식 명칭 + 별칭을 함께 입력, 관리자가 검토 후 승인.
> → **B안(유저가 별칭도 함께 제안)으로 확정**. Vivino도 동일 방식.
>
> **관리자 처리 방식 확정 — 3가지 액션**
> - **승인 (신규 등록)**: 새로운 술 → Alcohol + Alias 생성
> - **별칭으로 병합**: 이미 있는 술인데 다른 이름으로 요청됨 → 기존 Alcohol에 alias만 추가
> - **거절**: 장난, 부적절한 요청 등
> 병합 기능의 효과: 거절당한 유저도 "내 요청이 별칭으로 반영됨" → 긍정적 경험 + 검색 품질 자연 향상
>
> **AlcoholRequest 테이블 설계**
> - status: PENDING / APPROVED / MERGED / REJECTED
> - mergedToAlcoholId: 별칭 병합 시 어떤 Alcohol에 붙였는지 기록
> - 목록 조회 시 similarAlcohols 포함 (기존 DB 유사 술 자동 검색 → 관리자 판단 도움)

---

**Q6. 가입 직후 유저에게 어떤 첫 경험을 줄 건지? (온보딩) ✅ 확정**
> ✅ 확정 — 가입 직후 "첫 노트 작성하기" 흐름으로 유도 (B안)
> - 이 서비스의 핵심 가치는 "내 기록"이므로 처음부터 노트 작성을 각인
> - 백엔드 구현: 로그인 응답에 isFirstLogin 필드 추가 → 프론트에서 분기 처리
> - 보완책: 관리자가 초반 노트 10~20개 직접 작성해 피드 채워두기 (A안 병행)

---

**Q7. 별점을 정수(1~5)로 유지할지, 0.5단위로 할지? ✅ 확정**
> 0.5단위(0.5~5.0)로 확정. Vivino와 동일한 방식. DB 컬럼 DECIMAL(2,1), 코드 타입 BigDecimal (11회차에 Double → BigDecimal 전환 완료 — 부동소수점 오차로 0.5 단위 검증이 간헐적으로 뚫리는 버그 발견 후 재작성).

---

**Q8. 태그 품질을 어떻게 관리할 건지? ✅ 확정**
> 태그 자동완성에 사용 횟수 표시 방식으로 확정 (예: #위스키 (234)).
> 맛/향은 FlavorSuggestion 관리자 목록에서 **선택만 가능** (Vivino 방식) — NoteFlavor 중간 테이블로 구현 완료.
> 일반 태그는 자유입력 + 사용 횟수 표시.
> 사용 횟수는 NoteTag 개수 실시간 카운팅 방식 (Tag 테이블 컬럼 추가 없음).
> 맛/향을 선택 전용으로 한 이유: 자유 입력 시 "바닐라", "바닐라향", "vanilla" 등 같은 것이 다르게 저장되어 술 상세 페이지/Discovery 통계를 낼 수 없음.

---

**Q9. 프로필 페이지를 어떻게 구성할 건지? ✅ 확정**
> 팔로우 기능이 없는 상태에서 타인 프로필에 들어갔을 때 뭘 보여줄 건지.
> ✅ 확정 — 미니 포트폴리오 방식
> - 공개 노트 목록
> - 많이 마신 술 카테고리 (예: "위스키 애호가")
> - 평균 별점, 총 노트 수, 즐겨 쓰는 태그 Top 5
> - 별도 집계 테이블 없이 기존 데이터 쿼리로 구현

---

**Q10. 회원탈퇴 처리 방식을 어떻게 할 건지? ✅ 확정**
> ✅ 확정 (30일 유예 후 완전삭제 + 유예기간 중 연관 데이터 유지)
> - 탈퇴 시 deleted_at 기록 + 모든 노트 isPublic = false
> - 30일 유예 기간 후 Hard Delete 스케줄러로 완전 삭제

**Q20. N+1 쿼리 문제를 해결해야 하는가?** ⭐⭐⭐⭐⭐

> **문제 발견**
> NoteResponse.from(note)에서 note.getUser(), note.getAlcohol()을 호출함.
> 둘 다 FetchType.LAZY라서, 피드에서 노트 50개를 조회하면:
> - 노트 50개 쿼리 1번
> - 각 노트마다 user 조회 50번 + alcohol 조회 50번
> → 총 101번 쿼리 발생 (N+1 문제)
> 지금은 데이터가 적어서 체감이 없지만 피드에 노트가 100개만 되어도 응답이 느려짐.
>
> **왜 LAZY 로딩을 썼나**
> LAZY 로딩은 연관 데이터를 "실제로 쓸 때만" DB에서 가져오는 방식.
> 예: note.getUser()를 코드에서 안 부르면 users 테이블을 아예 조회 안 함.
> 단건 조회처럼 연관 데이터가 필요 없는 경우에는 EAGER보다 훨씬 효율적.
> 그래서 JPA 권장 설정이기도 함.
>
> **문제는 목록 조회**
> 단건 조회와 달리 목록 조회는 루프를 돌면서 각 note마다 getUser()를 호출하게 됨.
> 결국 노트 수만큼 쿼리가 터지는 구조 — LAZY 로딩의 의도치 않은 부작용.
>
> **왜 EAGER로 바꾸면 안 되나**
> EAGER로 바꾸면 note 하나를 조회할 때도 항상 user와 alcohol을 JOIN해서 가져옴.
> 예: 단건 조회, 존재 여부 확인 등 user가 불필요한 상황에서도 항상 JOIN 발생 → 오히려 더 비효율.
> "필요할 때 필요한 곳에서만" 해결해야 함.
>
> **해결: @EntityGraph**
> Repository의 목록 조회 메서드에만 @EntityGraph를 붙여서 해당 쿼리에서만 JOIN으로 한 번에 가져오도록 지정.
> LAZY 설정은 그대로 유지하면서, 목록 조회 시에만 동작 방식을 바꾸는 것.
> 결과: 101번 쿼리 → 1번으로 감소.
>
> **JOIN FETCH와의 차이**
> JOIN FETCH는 JPQL에 직접 작성하는 방식 — 더 유연하지만 쿼리를 직접 써야 함.
> @EntityGraph는 어노테이션으로 선언하는 방식 — 코드가 깔끔하고 Spring Data JPA 메서드 이름 쿼리와 함께 쓰기 좋음.
> 이 프로젝트에서는 @EntityGraph를 선택할 예정.
>
> **면접 예상 질문과 답변**
> Q: "왜 처음부터 EAGER로 안 했나요?"
> A: "EAGER는 항상 JOIN을 강제해서 불필요한 경우에도 부하가 생깁니다. LAZY를 기본으로 유지하고, 목록 조회처럼 연관 데이터가 반드시 필요한 곳에만 @EntityGraph로 한정해서 해결하는 게 더 정교한 접근입니다."

**Q21. 검색 품질 개선 — LIKE에서 Full-Text Search로** ⭐⭐⭐⭐⭐

> **현재 방식의 한계**
> `LIKE %키워드%` 방식은 두 가지 문제가 있음:
> 1. 성능: 앞에 %가 붙으면 인덱스를 못 써서 데이터가 많아질수록 전체 테이블을 스캔함
> 2. 품질: 모든 결과를 동등하게 취급 — "블랙라벨"을 검색했을 때 정확히 일치하는 술이나 별칭에 "블랙"이 포함된 술이나 순서가 같음. 어떤 게 먼저 나올지 예측 불가.
>
> **Full-Text Search가 다른 이유**
> DB가 각 단어를 쪼개서 별도의 인덱스(역색인)를 미리 만들어 놓는 방식.
> 예: "조니워커 블랙라벨" → "조니워커", "블랙라벨" 각각 인덱스 생성
> 검색 시 이 인덱스를 바로 조회하므로 LIKE보다 훨씬 빠름.
> 또한 각 결과에 relevance score(관련도 점수)를 부여 — 정확히 일치할수록 점수가 높아서 상위 노출 가능.
>
> **실제 서비스 비교**
> - Vivino: 와인 이름 정확 일치 > 생산자 이름 일치 > 부분 일치 순으로 정렬 + 리뷰 수 많은 와인 상위 노출
> - Untappd: 맥주 이름 정확 일치 우선 + 체크인 수 반영
> 둘 다 단순 포함 여부가 아니라 "얼마나 관련 있는가"로 정렬함.
>
> **전환 시점**
> 지금은 술 DB가 수백 개 수준이라 LIKE로도 충분함.
> 데이터가 수천 개를 넘거나 검색 결과 순서에 대한 불만이 생길 때 전환 예정.
> MySQL 8.0 이상에서는 별도 설치 없이 FULLTEXT 인덱스 추가만으로 바로 사용 가능.
>
> **면접 예상 질문과 답변**
> Q: "처음부터 Full-Text Search를 쓰지 않은 이유가 있나요?"
> A: "초반에는 술 DB가 작아서 LIKE로도 성능 문제가 없었고, 오버엔지니어링을 피하고 싶었습니다. 단순한 방식으로 시작해서 실제로 한계를 느끼는 시점에 전환하는 게 더 실용적이라고 판단했습니다. 전환 경로도 LIKE → MySQL Full-Text Search → 필요하면 Elasticsearch 순으로 단계적으로 계획해뒀습니다."

**Q22. 동시 편집 문제 — 낙관적 락(Optimistic Locking)** ⭐⭐⭐⭐

> **문제 발견**
> 같은 노트를 폰과 노트북에서 동시에 열고, 둘 다 수정 후 저장하면?
> 지금은 나중에 저장한 쪽이 먼저 저장한 내용을 조용히 덮어씀 (Last Write Wins).
> 유저 입장에서는 내가 분명히 저장했는데 내용이 사라진 상황 — 신뢰도 하락.
>
> **락(Lock)이란**
> 여러 곳에서 동시에 같은 데이터를 수정하려 할 때 충돌을 방지하는 메커니즘.
> 크게 두 가지 방식이 있음:
>
> - 비관적 락(Pessimistic Lock): "누군가 수정 중이면 아예 접근 못 하게 잠금"
>   → DB에 SELECT FOR UPDATE로 레코드 자체를 잠금. 다른 사람은 수정 완료될 때까지 대기.
>   → 충돌이 자주 일어나는 환경에 적합하지만, 대기 시간 발생 + 데드락 위험.
>
> - 낙관적 락(Optimistic Lock): "충돌이 드물다고 낙관적으로 가정하고, 저장할 때 확인"
>   → 잠금 없이 수정 허용. 저장 직전에 "내가 읽었을 때랑 지금이랑 같은 상태인가?" 확인.
>   → 다르면 충돌로 판단하고 에러 반환.
>
> **왜 낙관적 락을 선택하나**
> 노트 편집은 동시 충돌이 드문 케이스 — 같은 노트를 두 기기에서 동시에 수정하는 경우가 많지 않음.
> 비관적 락은 충돌이 거의 없는데도 항상 DB 잠금을 걸어서 성능 낭비.
> 낙관적 락은 평상시엔 아무 부하 없이 작동하다가 진짜 충돌이 생겼을 때만 처리함.
>
> **동작 방식 (@Version)**
> Note 엔티티에 version 컬럼 추가. 수정할 때마다 version이 1씩 올라감.
> 저장 시 "내가 읽었을 때 version"과 "현재 DB의 version"을 비교.
> 다르면 → 그 사이 누군가 수정했다는 뜻 → OptimisticLockException 발생 → "다른 곳에서 수정됨, 새로고침 필요" 안내.
>
> **면접 예상 질문과 답변**
> Q: "비관적 락과 낙관적 락의 차이를 알고 있나요? 왜 낙관적을 선택했나요?"
> A: "비관적 락은 충돌을 아예 막는 방식이고, 낙관적 락은 충돌을 감지해서 처리하는 방식입니다. 노트 편집은 같은 노트를 두 곳에서 동시에 수정하는 경우가 드물어서, 항상 DB 잠금을 거는 비관적 락은 오버헤드가 크다고 판단했습니다. 충돌 빈도가 낮고 사용자 경험(대기 없음)이 중요한 경우엔 낙관적 락이 더 적합합니다."

> "위스키"로 검색하는 유저가 100명이면 똑같은 쿼리를 100번 실행하는 셈.
>
> **캐싱이란**
> 자주 요청되는 데이터를 메모리(RAM)에 저장해두고, 같은 요청이 오면 DB까지 가지 않고 메모리에서 바로 꺼내주는 방식.
> DB 접근: 수십 ms / 메모리 접근: 수 μs (수천 배 빠름)
>
> **어떤 데이터에 적합한가**
> 캐싱은 읽기가 많고 변경이 적은 데이터에 효과적.
> - 적합: 술 정보, 맛/향 제안 목록(FlavorSuggestion), 카테고리 목록 → 거의 안 바뀜
> - 부적합: 노트 목록, 피드 → 실시간으로 계속 바뀜
>
> **캐시 무효화 문제**
> 캐싱의 핵심 어려움은 "언제 캐시를 지울 것인가"임.
> 술 정보가 수정됐는데 캐시가 그대로면 유저에게 옛날 정보를 보여주게 됨.
> 해결: 관리자가 술 정보를 수정/추가할 때 해당 캐시를 명시적으로 삭제(@CacheEvict).
> 지금은 관리자 API가 없으니 나중에 함께 구현 예정.
>
> **단계적 확장 계획**
> 1단계: Spring Cache (로컬 메모리 캐시) — 서버 1대일 때 충분
> 2단계: Redis 캐시 — 서버가 여러 대로 늘어나면 로컬 메모리 캐시는 서버마다 달라서 문제 발생. Redis는 별도 서버에서 공유 캐시 역할.
>
> **면접 예상 질문과 답변**
> Q: "캐시를 도입했을 때 생길 수 있는 문제는 무엇인가요?"
> A: "캐시 무효화 문제입니다. 원본 데이터가 바뀌었을 때 캐시가 그대로면 오래된 데이터를 보여주게 됩니다. 이를 해결하기 위해 술 정보 수정 시 @CacheEvict로 관련 캐시를 삭제하도록 구현했습니다. 또한 서버가 여러 대로 늘어나면 로컬 메모리 캐시는 서버마다 달라지는 문제가 있어서 Redis로 전환하는 것도 고려하고 있습니다."

**Q24. Refresh Token Rotation + Reuse Detection을 왜 도입했나요?** ⭐⭐⭐⭐⭐ ✅ 적용 완료

> **문제 발견**
> reissue 엔드포인트에서 기존 Refresh Token(RT)을 hard delete한 뒤 새 RT를 발급하는 구조였음.
> "토큰을 지우고 새걸 주니까 안전하다"고 생각했는데, 실제로는 **탈취 감지가 불가능한** 구조였음.
>
> 공격자가 RT1을 탈취해서 먼저 reissue를 호출하면:
> 1. RT1 삭제 → 공격자에게 RT2 발급
> 2. 정상 유저가 RT1로 reissue 시도 → `INVALID_TOKEN` 에러만 반환
> 3. 정상 유저는 "토큰 만료됐나보다"하고 재로그인, 공격자는 RT2로 계속 활동
>
> 탈취 사실을 서버가 전혀 눈치채지 못하고, 공격자는 재로그인 전까지 자유롭게 돌아다님.
>
> **배경 지식 — OAuth 2.0 Security BCP**
> RFC 6819와 draft-ietf-oauth-security-topics에 정의된 Refresh Token Rotation with Reuse Detection 패턴.
> 핵심 아이디어: "이미 사용된(revoked) 토큰이 다시 들어오면 → 누군가 탈취해서 재사용하는 것" 으로 판단.
>
> **해결**
> - RT 엔티티에 `revoked` boolean 필드 추가
> - 정상 reissue: hard delete 대신 `revoked=true` 표시 + 새 RT 발급 (흔적을 남김)
> - revoked 토큰이 재사용되면: **해당 유저의 모든 RT 삭제** (공격자가 보유한 새 RT2도 함께 무효화)
> - 정상 유저는 재로그인 필요하지만, 공격자는 완전 차단됨
>
> 부수 효과로 revoked 토큰이 DB에 누적되므로 주기적으로 만료/revoked 토큰을 정리하는 스케줄러가 필요 — 추후 구현 예정.
>
> **면접 예상 질문과 답변**
> Q: "기존 hard delete 방식과 무엇이 다른가요?"
> A: "hard delete는 사용된 토큰을 바로 지워버리기 때문에 '누가 먼저 사용했는지' 알 방법이 없습니다. revoke 방식은 흔적을 남기기 때문에, 이미 처리된 토큰이 다시 들어오면 재사용으로 판단할 수 있습니다. 이 차이가 탈취 감지의 핵심입니다."
>
> Q: "이 문제를 어떻게 발견했나요?"
> A: "탈취 시나리오를 직접 시뮬레이션해보면서 발견했습니다. '공격자가 토큰을 먼저 사용하면 정상 유저는 어떻게 되지?'를 따라가보니, 서버 입장에서는 두 번의 reissue 요청이 구분되지 않는다는 걸 깨달았습니다. 그래서 표준 스펙을 찾아봤고 OAuth 2.0 Security BCP에 이 패턴이 정의돼 있다는 걸 알게 됐습니다."

**Q25. 별점을 Double에서 BigDecimal로 바꾼 이유가 무엇인가요?** ⭐⭐⭐⭐ ✅ 적용 완료

> **문제 발견**
> 별점 0.5 단위 검증 로직이 `Math.round(rating * 10) % 5 != 0` 방식이었음.
> Double이 IEEE 754 부동소수점 타입이라 `3.5`를 정확히 담지 못하고 `3.500000000001` 같은 값이 될 수 있음.
> 이 경우 `3.500000000001 * 10 = 35.00000000001` → `Math.round → 35` → `35 % 5 == 0` 으로 통과.
> 반대로 `3.4999...` 같은 케이스는 `Math.round → 34` → `34 % 5 != 0` 으로 거부될 수도 있음.
> **정상적인 3.5 입력이 검증을 통과하거나 실패하는 게 환경에 따라 달라지는** 불안정한 구조.
>
> **배경 지식**
> Double은 "약 15자리 정밀도"의 근사값을 다루는 타입. 금액·수량·별점처럼 정확한 단위가 있는 수치에는 적합하지 않음.
> BigDecimal은 정확한 십진수 연산을 보장. 실무에서 금액/포인트/평점 같은 값은 BigDecimal을 쓰는 게 표준.
> 또한 DB 컬럼이 `DECIMAL(2,1)`인데 Java가 Double이면, DB는 정밀도를 보장하는데 애플리케이션 레이어에서 오차가 생기는 타입 불일치 문제도 있었음.
>
> **해결**
> - 엔티티·DTO·서비스 전체 rating 필드를 BigDecimal로 교체
> - 검증 로직을 `rating.remainder(new BigDecimal("0.5")).compareTo(BigDecimal.ZERO) != 0` 방식으로 재작성
> - `@Min/@Max` → `@DecimalMin/@DecimalMax`로 교체
> - 허용 범위 1.0~5.0 → 0.5~5.0 조정 (0.0은 "미평가"와 혼동 방지)
>
> **면접 예상 질문과 답변**
> Q: "Double 대신 BigDecimal을 써야 하는 경우가 어떤 경우인가요?"
> A: "정확한 십진수 연산이 필요한 경우입니다. 금액, 수량, 별점처럼 0.1이나 0.5 같은 단위가 의미를 가지는 값이 해당됩니다. Double은 내부적으로 이진수 근사값이라서 '정확히 3.5'를 보장하지 못합니다. 반면 BigDecimal은 문자열 기반으로 정확한 십진수를 다루기 때문에 이런 케이스에 적합합니다."
>
> Q: "이 버그를 어떻게 발견했나요?"
> A: "DB 컬럼이 DECIMAL인데 Java 타입이 Double인 걸 보면서 '왜 다르지?'라는 의문이 생겼습니다. 찾아보니 Double의 부동소수점 특성 때문에 0.5 단위 검증에서 오차가 생길 수 있다는 걸 알게 됐고, 기존 Math.round 방식을 직접 시뮬레이션해보니 3.5001 같은 값이 조용히 통과한다는 걸 확인했습니다."

**Q26. 중복 신고 방지에 DB unique 제약을 추가한 이유가 무엇인가요?** ⭐⭐⭐ ✅ 적용 완료

> **문제 발견**
> 중복 신고 방지 로직이 `existsByReporterIdAndNoteId` 체크 후 `save`하는 패턴이었음.
> 그런데 Report 엔티티에는 DB unique 제약이 없었음.
> 두 요청이 거의 동시에 도착하면:
> 1. 요청A: exists 체크 → false (아직 없음)
> 2. 요청B: exists 체크 → false (아직 없음)
> 3. 요청A: save → 신고 저장
> 4. 요청B: save → **중복 신고 저장** (DB 제약이 없으니 막지 못함)
>
> 이를 TOCTOU(Time-Of-Check To Time-Of-Use) race condition이라고 부름.
> 같은 유저가 신고 버튼을 연타하면 신고 수를 부풀리는 어뷰징이 가능한 상태였음.
>
> **배경 지식**
> 애플리케이션 레벨 체크(`existsBy`)는 단일 스레드에서 순차 실행될 때만 신뢰할 수 있음.
> 동시 요청이 들어오면 둘 다 "없다"는 결과를 받고 둘 다 진행하게 됨.
> DB unique 제약은 트랜잭션 수준에서 원자적으로 강제되므로, 동시 요청이 와도 한 쪽만 성공함.
> **DB 제약을 진실의 원천(source of truth)으로 두는 것**이 정합성 보장의 정석.
>
> **해결**
> - Report 엔티티에 `(reporter_id, note_id)` 복합 unique 제약 추가
> - `save()`를 `try/catch`로 감싸 `DataIntegrityViolationException` → `ALREADY_REPORTED(409)` 비즈니스 예외로 변환
> - 애플리케이션 레벨 `existsBy` 검증은 정상 케이스의 빠른 에러 응답용 UX 계층으로 유지
> - 결과: 앱 레벨(UX) + DB 제약(정합성) 이중 구조
>
> **면접 예상 질문과 답변**
> Q: "서비스 레벨에서 existsBy 체크를 이미 하고 있는데 DB 제약이 왜 필요한가요?"
> A: "동시 요청이 들어오면 existsBy 체크를 두 요청이 동시에 통과할 수 있습니다. DB 제약은 트랜잭션 레벨에서 원자적으로 처리되므로, 동시 요청이 와도 정확히 한 건만 저장됩니다. 서비스 레벨 체크는 '이미 신고했어요'를 빠르게 알려주는 UX용이고, DB 제약이 실제 정합성을 보장하는 역할입니다."
>
> Q: "DataIntegrityViolationException을 catch하는 방식에 단점이 있지 않나요?"
> A: "예외로 흐름을 제어하는 것이라 성능상 약간의 비용은 있습니다. 하지만 중복 신고 자체가 정상 케이스가 아니라 드물게 발생하는 예외 상황이기 때문에 실질적인 영향은 미미합니다. 오히려 동시성 문제를 DB가 보장해준다는 안전성이 더 중요하다고 판단했습니다."

**Q27. NoteStatus와 isPublic을 왜 분리해서 관리하나요?** ⭐⭐⭐ ✅ 적용 완료

> **문제 발견**
> "DRAFT면 비공개, PUBLISHED면 공개" 라는 단순한 생각으로 두 필드를 엮었음.
> 구체적으로는: createNote에서 isPublic을 항상 false로 고정, updateNote에서 DRAFT+isPublic=true 조합을 금지.
> 그 결과 **publishNote()가 status만 PUBLISHED로 바꾸고 isPublic은 건드리지 않기 때문에**,
> 발행을 해도 isPublic이 여전히 false → 공개 피드 쿼리(`status=PUBLISHED AND isPublic=true`)를 통과하는 노트가 없음.
> 공개 노트를 만드는 경로 자체가 없어진 상태였음.
>
> **배경 지식**
> `status`와 `isPublic`은 의미적으로 완전히 다른 차원의 필드다:
> - `NoteStatus (DRAFT/PUBLISHED)`: **라이프사이클** — "아직 작성 중인가, 발행됐는가"
> - `isPublic`: **공개 의도(intent)** — "다른 사람이 볼 수 있는가"
>
> 네이버 블로그·티스토리·Medium 등 실제 서비스를 조사해보니, 대부분 임시저장 단계에서 공개 여부를 **함께 설정**하는 방식이었음.
> DRAFT 상태에서 isPublic=true인 데이터가 존재해도, 피드 쿼리가 `status=PUBLISHED` 조건을 함께 걸기 때문에 실제 노출 위험은 없음.
>
> **해결**
> - isPublic 필드를 DTO에 복원, createNote에서의 하드코딩 제거
> - DRAFT+isPublic=true 금지 제약 제거
> - publishNote는 status만 바꾸고 isPublic은 사용자 설정값 그대로 유지
>
> **면접 예상 질문과 답변**
> Q: "DRAFT 상태에서 isPublic=true인 데이터가 DB에 있으면 문제가 없나요?"
> A: "피드 조회 쿼리가 status=PUBLISHED AND isPublic=true 조건을 함께 사용하기 때문에 DRAFT 노트는 아무리 isPublic=true여도 피드에 노출되지 않습니다. DB에 '의도적으로 일관성 없는 데이터'가 있는 게 아니라, 두 필드가 다른 의미를 가지는 것이기 때문에 문제가 없습니다."
>
> Q: "이 버그를 어떻게 발견했나요?"
> A: "공개 피드를 테스트하다가 노트가 하나도 안 나오는 걸 발견했습니다. publishNote API를 호출했는데도 피드에 안 뜨길래 쿼리 조건을 역으로 따라가보니, status는 PUBLISHED가 됐는데 isPublic은 여전히 false였습니다. 코드를 추적하니 isPublic=true로 만들 수 있는 경로가 코드 상에 존재하지 않는다는 걸 알게 됐습니다."
> A: "캐시 무효화 문제입니다. 원본 데이터가 바뀌었을 때 캐시가 그대로면 오래된 데이터를 보여주게 됩니다. 이를 해결하기 위해 술 정보 수정 시 @CacheEvict로 관련 캐시를 삭제하도록 구현했습니다. 또한 서버가 여러 대로 늘어나면 로컬 메모리 캐시는 서버마다 달라지는 문제가 있어서 Redis로 전환하는 것도 고려하고 있습니다."
